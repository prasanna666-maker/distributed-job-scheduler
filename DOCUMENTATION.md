## 1. System Architecture

### Components & Flow Diagram
```
  ┌─────────────────────────────────────────────────────────┐
  │                    React Frontend                       │
  │  (Dashboard, Job Explorer, Queue Config, Metrics)       │
  └─────────────────────┬───────────────────────────────────┘
                        │ REST API (JSON)
  ┌─────────────────────▼───────────────────────────────────┐
  │               Spring Boot Backend                       │
  │  ┌──────────┐ ┌──────────┐ ┌───────────┐ ┌───────────┐  │
  │  │ Auth     │ │ Job      │ │ Queue     │ │ Scheduler │  │
  │  │ (JWT)    │ │ Service  │ │ Service   │ │ Service   │  │
  │  └──────────┘ └──────────┘ └───────────┘ └───────────┘  │
  │  ┌──────────────────────────────────────────────────┐   │
  │  │            Worker Service (embedded)              │   │
  │  │  Poll → Claim (FOR UPDATE SKIP LOCKED) → Execute │   │
  │  │  Heartbeat thread │ Graceful shutdown hook        │   │
  │  └──────────────────────────────────────────────────┘   │
  └─────────────────────┬───────────────────────────────────┘
                        │ JPA / Hibernate
  ┌─────────────────────▼───────────────────────────────────┐
  │                     MySQL 8.x                           │
  │  (InnoDB, row-level locking, SKIP LOCKED support)       │
  └─────────────────────────────────────────────────────────┘
```

The system is designed as a modular monolith:
1. **Interactive Client**: React SPA that displays metrics charts and manages queues.
2. **REST API**: Spring Boot controllers using Spring Security (JWT) for authentication.
3. **Execution Engine**: Embedded concurrent worker daemon polling tasks using database-backed queues.
4. **State Machine**: Enforces rigid transitions across job states.
5. **Persistence**: MySQL 8.x executing row-level locks on unclaimed items.

## 2. Database Schema (16 Tables)

Below is the complete entity list mapped directly to InnoDB tables:

1. **`users`**: Interactive dashboard credentials.
2. **`organizations`**: Tenants/owners of projects.
3. **`organization_members`**: Joins users to organizations with roles (`OWNER`, `ADMIN`, `MEMBER`).
4. **`projects`**: Scope namespaces within an organization.
5. **`retry_policies`**: Holds retry backoff algorithms and delay configurations.
6. **`queues`**: Priority channels with custom concurrency limits and reference retry policies.
7. **`jobs`**: The main task queue. Holds state, metadata payload, scheduled times, and execution attempts.
8. **`workers`**: Live registries of running job scheduler nodes.
9. **`job_executions`**: History logs of every single job execution attempt.
10. **`worker_heartbeats`**: Raw CPU/Memory/Active jobs stats reported by workers.
11. **`job_logs`**: Text logs generated by the workers during job execution.
12. **`scheduled_jobs`**: Templates containing cron expressions to spawn periodic jobs.
13. **`dead_letter_queue`**: Quarantine table holding permanently failed jobs.
14. **`api_keys`**: Cryptographically hashed keys used by workers/machines to access endpoints.
15. **`audit_logs`**: Immutable security ledger tracking all state mutations.
16. **`webhooks`**: Real-time HTTP event notification settings.

## 3. Concurrency Safety & SKIP LOCKED

The core job claiming query guarantees atomic operations under high load without double-claims:

```sql
SELECT * FROM jobs
WHERE queue_id = ?
  AND status = 'QUEUED'
  AND (scheduled_at IS NULL OR scheduled_at <= NOW())
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

### Concurrency Guarantees
1. **`FOR UPDATE`**: Locks the matched row exclusively.
2. **`SKIP LOCKED`**: Skips any rows locked by other concurrent queries instead of waiting.
3. **Index optimization**: Uses a composite index `(queue_id, status, priority DESC, scheduled_at)` to look up candidates in $O(1)$ time, eliminating table scans.

## 4. REST API Endpoint List

### Authentication
* `POST /api/auth/register` — Create user
* `POST /api/auth/login` — Returns JWT token

### Organizations & Projects
* `POST /api/organizations` — Register new tenant organization
* `POST /api/organizations/{orgId}/projects` — Add project under organization
* `POST /api/projects/{projectId}/queues` — Create queue under project

### Job Operations
* `POST /api/queues/{queueId}/jobs` — Submit immediate/delayed job
* `POST /api/queues/{queueId}/jobs/batch` — Bulk submit up to 100 jobs
* `GET /api/queues/{queueId}/jobs` — Paginated list of jobs (supports status filter)
* `GET /api/jobs/{jobId}` — View detailed job state and metadata
* `POST /api/jobs/{jobId}/requeue` — Re-run a dead letter job

## 5. Design Decisions

* **Database as Queue**: Chosen to reduce infrastructure overhead. MySQL 8.x `SKIP LOCKED` scales to ~10,000 operations/sec.
* **Separation of Concern**: Embedded worker runs inside a separate thread pool (`ExecutorService`). If the worker crashes, the system transitions the job to `FAILED` and requeues it via retry policies.
* **Graceful Shutdown**: The worker listens to Spring Context destroy hooks, completes current in-flight runs, and marks itself `OFFLINE` before shutting down the executor.
